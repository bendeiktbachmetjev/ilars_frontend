# Публикация ABBA LARS в Google Play Store — пошаговая инструкция

Пошаговое руководство: что настраивать, куда нажимать и что писать при публикации приложения в Google Play.

---

## Часть 1. Подготовка перед публикацией ✅ (настроено)

### 1.1. Ключ для подписи (keystore)

- **Keystore:** `android/app/upload-keystore.jks` (уже есть; делайте резервную копию).
- **Подпись:** используется файл `android/key.properties` (не коммитится; см. `key.properties.example`).

Текущая схема: при сборке `flutter build appbundle --release` Gradle читает `android/key.properties`. Если файла нет — используются переменные окружения `RELEASE_STORE_*` или `gradle.properties` (секреты из `gradle.properties` убраны).

**Если будете настраивать на другом компьютере:** скопируйте `key.properties.example` в `key.properties`, укажите пароли и убедитесь, что `upload-keystore.jks` лежит в `android/app/`.

### 1.2. Сборка релизного AAB (Android App Bundle)

Google Play принимает формат **AAB**, не APK.

```bash
cd /путь/к/frontend/frontend
flutter build appbundle --release
```

Готовый файл: **`build/app/outputs/bundle/release/app-release.aab`** — его загружаете в Play Console.

Удобно собирать скриптом: из папки `frontend/frontend` выполните **`./play_store/build_release.sh`**.

### 1.3. Версия в pubspec.yaml

Перед каждой новой загрузкой в Play увеличивайте версию в `pubspec.yaml`:

```yaml
version: 1.1.0+3   # 1.1.0 — версия для пользователя, 3 — versionCode (целое, только вверх)
```

При следующем релизе, например: `1.2.0+4`. В Play Console нельзя загрузить сборку с тем же или меньшим `versionCode`, чем у уже загруженной.

### 1.4. Что подготовить заранее (готовые файлы в проекте)

В папке **`play_store/`** лежит всё, что можно скопировать или выложить:

| Файл | Назначение |
|------|------------|
| **store_listing_copy.txt** | Название, краткое (80 симв.) и полное описание — копировать в Store listing. |
| **privacy_policy.html** | Готовая страница политики конфиденциальности. Выложите на любой хостинг (сайт ABBA, GitHub Pages и т.д.) и укажите полученный URL в Play Console. |
| **app_content_answers.md** | Подсказки для разделов Data safety, Health declaration, Ads, Content rating, Target audience. |
| **build_release.sh** | Скрипт сборки AAB: запуск из `frontend/frontend`: `./play_store/build_release.sh`. |

Дополнительно нужно подготовить (вручную):

- **Иконка приложения** — 512×512 px (PNG, без прозрачности). Можно экспортировать из `android/app/src/main/res/mipmap-*` или из дизайна.
- **Feature graphic** — 1024×500 px (баннер в карточке приложения).
- **Скриншоты** — минимум 2, лучше 4–8. Телефон: 1080×1920 или 1080×2340. Сделайте с эмулятора или устройства после входа по тестовому коду.

## Часть 2. Аккаунт разработчика Google Play

### 2.1. Регистрация

1. Перейдите на [Google Play Console](https://play.google.com/console).
2. Войдите в Google-аккаунт (лучше корпоративный для приложения ABBA).
3. При первом входе примите соглашение с разработчиком.
4. Оплатите единоразовый регистрационный взнос (на момент написания — около $25). Оплата — в разделе настроек аккаунта (иконка шестерёнки → Payment profile).

### 2.2. Создание приложения

1. В Play Console на главной нажмите **«Создать приложение»** (Create app).
2. Заполните:
   - **Название приложения:** ABBA LARS
   - **Язык по умолчанию:** например, English (United States) или нужный вам.
   - **Тип приложения:** Приложение (App) или Игра (Game) — выберите «Приложение».
   - **Бесплатное или платное:** выберите бесплатное/платное по факту.
3. Подтвердите заявления (например, о соблюдении правил, экспорте и т.д.) и нажмите **«Создать приложение»**.

После этого откроется панель приложения с боковым меню.

## Часть 3. Заполнение карточки приложения (Store listing)

В боковом меню: **«Наличие в магазине» (Release)** → **«Основная страница магазина» (Main store listing)** (или **«Store presence» → «Main store listing»**).

### 3.1. Основные данные

Готовые тексты — в **`play_store/store_listing_copy.txt`** (копируйте оттуда).

- **Название приложения:** ABBA LARS (до 30 символов).
- **Краткое описание:** до 80 символов (в файле уже есть вариант).
- **Полное описание:** до 4000 символов (полный текст в том же файле).

### 3.2. Графика

- **Иконка приложения 512 x 512:** загрузите PNG 512×512.
- **Feature graphic 1024 x 500:** загрузите баннер.
- **Скриншоты телефона:** минимум 2, максимум 8. Размеры: минимум 320 px по короткой стороне, максимум 3840 px по длинной. Обычно делают 1080×1920 или 1080×2340.

При необходимости можно добавить скриншоты для 7" и 10" планшетов в соответствующих блоках.

### 3.3. Классификация

- **Категория приложения:** например, «Здоровье и фитнес» (Health & Fitness).
- **Теги (опционально):** по желанию для поиска.

### 3.4. Контакты

- **Электронная почта:** контакт для пользователей и для Google (поддержка).
- **Телефон** и **веб-сайт** — по желанию.

### 3.5. Сохранение

Внизу страницы нажмите **«Сохранить»** (Save). Можно сохранять по частям; главное — не оставлять обязательные поля пустыми перед отправкой на модерацию.

## Часть 4. Политика конфиденциальности и объявления

### 4.1. Политика конфиденциальности

- В разделе **«Политика конфиденциальности»** укажите **URL** на публичную страницу политики.
- Готовая страница: **`play_store/privacy_policy.html`**. Выложите её на сайт ABBA или на GitHub Pages (см. ниже), затем вставьте полученный URL в Play Console.

**Как выложить на GitHub Pages:** создайте репозиторий (или папку `docs/` в существующем), положите туда `privacy_policy.html` как `index.html` в корне страницы, включите Pages в настройках репозитория (Settings → Pages → Source: main branch / docs). URL будет вида `https://username.github.io/repo-name/`.

### 4.2. Объявления (Ads)

- Если в приложении **нет рекламы** — в разделе про рекламу (например, «Ads declaration») укажите, что приложение не содержит рекламы.

## Часть 5. Контент приложения (App content)

В боковом меню откройте **«Контент приложения» (App content)** или **«Policy» → «App content»**. Там несколько обязательных пунктов.

### 5.1. Политика конфиденциальности

- Уже указана выше — ссылка на политику.

### 5.2. Health & Fitness (данные о здоровье)

- Заполните форму **«Health and fitness»**: приложение собирает данные о здоровье (шаги, симптомы, шкалы LARS/CRADI и т.д.).
- Точные формулировки — в **`play_store/app_content_answers.md`** (раздел Health and fitness).

### 5.3. Рейтинг контента (Content rating)

1. Нажмите **«Content rating»**.
2. Нажмите **«Начать анкету»** (Start questionnaire) или **«Create rating»**.
3. Укажите email и выберите категорию приложения (например, «Health, Medical»).
4. Ответьте на вопросы анкеты (насилие, контент для взрослых и т.д.). Для медицинского/оздоровительного приложения без соцсетей и покупок в приложении обычно получается рейтинг «Для всех» или «3+».
5. Сохраните рейтитинг и при необходимости привяжите его к приложению.

### 5.4. Целевая аудитория (Target audience)

1. Откройте **«Target audience and content»** или **«Target audience»**.
2. Укажите возрастные группы (например, 18+ для медицинского приложения).
3. Сохраните.

### 5.5. Data safety (Безопасность данных)

1. В **«App content»** откройте **«Data safety»** → **«Start»** / **«Заполнить»**.
2. Готовые подсказки и формулировки — в **`play_store/app_content_answers.md`** (что отмечать, какие типы данных, краткий текст для блока).
3. Кратко: данные собираются (да); типы — Health, при необходимости App activity / Device or other IDs; цель — App functionality; данные не продаются; политика конфиденциальности — ваш URL.
4. Сохраните форму.

### 5.6. Практики по обеспечению безопасности приложений (App access)

- Если есть закрытые зоны (например, только по коду от врача), укажите это и при необходимости добавьте тестовый доступ для ревьюеров.

### 5.7. Реклама (Ads)

- Если рекламы нет — отметьте, что приложение без рекламы.

## Часть 6. Загрузка сборки (Release)

### 6.1. Выбор трека

В боковом меню: **«Release»** → **«Production»** (или сначала **«Testing»** → **«Internal testing»**).

- **Internal testing** — до 100 тестеров по email, быстрая проверка.
- **Closed testing** — ограниченная группа.
- **Open testing** — открытая бета.
- **Production** — публичный релиз в магазине.

Для первого раза можно загрузить в **Internal testing**, проверить, что всё отображается, затем перевести в **Production**.

### 6.2. Создание релиза

1. Нажмите **«Create new release»** (Создать новый выпуск).
2. Если появится предложение подписать приложение через Google (Play App Signing) — согласитесь. Загружайте свой AAB; ключ подписи для распространения может остаться у Google (рекомендуется).
3. В блоке **«App bundles»** нажмите **«Upload»** (Загрузить) и выберите файл:
   `build/app/outputs/bundle/release/app-release.aab`
4. После загрузки укажите **Release name** (например, «1.1.0 (3)») и при необходимости **Release notes** (что нового — для пользователей).

### 6.3. Сохранение и отправка

1. Нажмите **«Save»** (Сохранить).
2. Нажмите **«Review release»** (Проверить выпуск). Система покажет предупреждения, если чего-то не хватает (скриншоты, политика, рейтинг и т.д.).
3. Если всё зелёное — нажмите **«Start rollout to Production»** (или «Start rollout to Internal testing» для теста). Подтвердите.

После этого релиз уходит на проверку. Для Production модерация обычно занимает от нескольких часов до нескольких дней.

---

## Часть 7. Что проверять перед каждой новой загрузкой

- Увеличена версия и `versionCode` в `pubspec.yaml`.
- Собран новый AAB: `flutter build appbundle --release`.
- Подпись выполняется тем же keystore, что и раньше.
- В **Release** загружен новый AAB, указаны release name и release notes.
- Все обязательные разделы в **App content** и **Store listing** заполнены и сохранены.

## Краткий чек-лист

| Шаг | Где | Действие |
|-----|-----|----------|
| 1 | Локально | Создать keystore, настроить key.properties или переменные окружения |
| 2 | Локально | `flutter build appbundle --release` |
| 3 | Play Console | Оплатить регистрацию, создать приложение |
| 4 | Store listing | Название, краткое/полное описание, иконка 512, feature graphic 1024×500, скриншоты |

После прохождения модерации приложение появится в Google Play по ссылке вида:  
`https://play.google.com/store/apps/details?id=com.abba.lars`

Ваш applicationId: **com.abba.lars** — эта ссылка будет актуальна после публикации.
